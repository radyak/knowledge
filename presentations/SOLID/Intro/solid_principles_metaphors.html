Single Responsibility Principle

The Busy Bee
Just like how a worker bee has one job - collecting nectar - each class should have only one reason to change and one responsibility to fulfill.
A class should have only one job. If a class has multiple responsibilities, changes to one responsibility might affect the others, making the code fragile and hard to maintain.

The Expandable House
Like adding rooms to a house without tearing down the foundation - you can extend functionality without modifying existing code.
Classes should be open for extension but closed for modification. You should be able to add new features by extending classes, not by changing existing code.

Liskov Substitution Principle
ü¶Ü
The Duck Family
Any duckling should be able to swim, quack, and waddle just like mama duck. Subclasses must be substitutable for their parent classes.
Objects of a superclass should be replaceable with objects of its subclasses without breaking the application. Child classes should strengthen, not weaken, the parent's contract.

The Restaurant Menu
Instead of one huge menu, have separate menus for breakfast, lunch, and dinner. Clients shouldn't depend on interfaces they don't use.
Many client-specific interfaces are better than one general-purpose interface. Don't force classes to implement methods they don't need.

Dependency Inversion Principle
üîå
The Universal Plug
Your lamp doesn't care about the power plant - it just needs a standard outlet. Depend on abstractions, not concrete implementations.
High-level modules shouldn't depend on low-level modules. Both should depend on abstractions. This makes your code more flexible and testable.
